Why Gas Town Works

The mathematics of multi-agent orchestration — a pragmatic cheat sheet

Gas Town · Multi-agent orchestrator for Claude Code
by Steve Yegge · github.com/steveyegge/gastown
Mathematical analysis · March 2026
Core claim: Gas Town's architecture works because it encodes three conservation laws — work is never silently lost, agent identity survives session death, and the main branch stays coherent. These aren't engineering heuristics. They're structural invariants. When they hold, you can safely parallelize. When they break, the system fails in predictable ways. Understanding why unlocks confident scaling.
1

Conserve Work — Nothing Gets Lost

Noether's theorem · time-translation symmetry

Every bead (task) lives in git. It can change state — open, hooked, in-progress, done — but it cannot silently vanish. Crashes, restarts, context window death: the work persists.

Math intuition: When a system's rules don't change over time, something is automatically conserved. Here, the rules for bead evolution are time-invariant → total work is always accounted for, like energy in physics.

→ File beads for everything. If it's not a bead, it can be forgotten.
2

Separate Identity from Session

Gauge invariance · persistent vs. ephemeral state

Sessions are cattle — expendable, interchangeable. Agent identity (role, history, inbox) is the conserved quantity that survives across sessions. The context window is the gauge-dependent part; the Beads-backed identity is gauge-invariant.

Math intuition: In physics, measurable reality doesn't depend on your coordinate system. Similarly, meaningful work state shouldn't depend on which session you're in. gt handoff changes the session (coordinates); identity persists (physics).

→ Use gt handoff liberally. Kill sessions without fear. Trust the identity layer.
3

Parallelize via Isolation, Serialize at Merge

Symmetric monoidal category · parallel composition ⊗

Each Polecat works in its own git worktree: full isolation. This makes parallel work well-defined — agents compose by working side by side (⊗), not in sequence (∘). The Refinery serializes at merge time, which is the only point where ordering matters.

Math intuition: Two operations can run in any order if they don't interfere. Git worktrees guarantee non-interference. The merge queue is the "coherence constraint" that ensures the final result is the same regardless of which polecat finished first.

→ More worktrees = more safe parallelism. Merge conflicts = interference you need to fix, not random chaos.
4

Don't Re-Boundary the Boundary

∂² = 0 · cohomological constraint

The Mayor sets the boundary: project plan, task decomposition, architectural decisions. Polecats and Crew work within that boundary. If the Mayor (or Overseer) redundantly re-specifies context already established, agents get contradictory signals and waste tokens rediscovering what was already settled.

Math intuition: In topology, taking the boundary of a boundary always gives you zero — there's nothing new there. Re-specifying established context is the informational equivalent: it adds noise, not signal. The "murderous Deacon" bug was exactly this — it re-bounded what the Mayor had already bounded.

→ Set context once at the Mayor level. In workers, reference beads — don't re-explain the whole project.
5

Get Coherence Right and Everything Else Follows

Mac Lane's coherence theorem · "if the basic diagrams commute, ALL diagrams commute"

You don't need to micromanage every agent interaction. You need to get the fundamental protocols right: GUPP (work on your hook = you must run it), handoff (serialize structural context, not just diffs), and merge (Refinery lands clean code on main). If these basic interactions are correct, the global system behavior is coherent regardless of the specific order or timing of agent actions. This is why Gas Town "degrades gracefully" — removing a role (Deacon, Witness) restricts the system to a smaller set of interactions where the coherence conditions still hold. You lose capability, not correctness.

Math intuition: Mac Lane proved that in certain algebraic structures, if a small set of canonical diagrams "commute" (give the same result regardless of path), then every possible diagram commutes. You don't have to check them all — just the generators. For Gas Town: nail GUPP + handoff + merge → all possible multi-agent workflows are coherent for free.

→ When things go wrong, check the three protocols — not individual agent behavior. Fix the generator, not every diagram.

⚠ When Symmetry Breaks

Every Gas Town failure maps to a broken conservation law. Merge conflicts = repository integrity violated. Lost work = bead not persisted. Agents stepping on each other = identity confusion or worktree contamination. Diagnosis: ask "which conservation law broke?" not "which agent misbehaved?"

↑ Scaling: Crew → Polecats → Convoys

Use Crew for design-heavy, iterative work (low parallelism, high context). Graduate to Polecats for well-defined, parallelizable tasks (high parallelism, low context). Convoys batch polecats against bead lists. Match the decomposition to the work's inherent structure — don't force parallelism on serial dependencies.

∞ The Wasteland Direction

Federating multiple Gas Towns together (the Wasteland) is where higher category theory kicks in. Trust levels between towns are morphisms-between-morphisms (2-categorical). The Wanted Board is a colimit over participating towns. This is genuinely new mathematical territory for distributed agent systems.
