The Mathematics of Multi-Agent Orchestration

An explorable explanation of why Gas Town works — March 2026

Gas Town is Steve Yegge's multi-agent orchestrator for Claude Code. It coordinates 20–30 parallel AI coding agents through persistent work tracking backed by Git. It sounds like chaos. But it works — and the reason it works is mathematical.

Underneath the Mad Max theming and the tmux panes, Gas Town encodes a small number of structural invariants — conservation laws, symmetry principles, and coherence conditions — that guarantee safe parallelism. If you understand these invariants, you can predict exactly when the system will succeed and when it will break.

Let's explore each one, hands-on.

I. Conservation of Work

The most fundamental property of Gas Town is that work cannot be silently lost. Every task is tracked as a "bead" — a structured issue stored as JSON in Git. A bead can change state — open, hooked, in-progress, done — but it can never quietly disappear.

In physics, Emmy Noether proved that every continuous symmetry of a system produces a conserved quantity. When the laws of physics don't change over time, energy is conserved. In Gas Town, the rules for how beads evolve are time-invariant — they don't depend on when you look at them. The consequence: total work is always accounted for.

In the simulation below, colored circles are beads (tasks) assigned to agents (the larger rings). Try clicking an agent to "crash" its session. Watch what happens to the beads — they scatter, but they never vanish. The counter at top never decreases.

Click an agent ring or use the button to crash a session. Beads persist — they float free until re-assigned.
Practical takeaway File beads for everything. If it's not a bead, it can be forgotten when a session dies. If it is a bead, it's conserved by construction — Git guarantees it.

II. Identity Survives Session Death

In physics, a gauge transformation changes your coordinate system without changing the underlying reality. The electric field you measure depends on your reference frame, but the charge of a particle does not. The charge is gauge-invariant — it's the real, physical thing.

Gas Town makes the same distinction. Each agent has two layers: an ephemeral session (the Claude Code context window — expendable, routinely killed) and a persistent identity (role bead + agent bead + work history + inbox — stored in Git, survives indefinitely).

The command gt handoff enacts exactly this: it serializes the structural context from the dying session into the persistent identity layer, then starts a fresh session that reads it back. The session (gauge-dependent quantity) changes; the identity (gauge-invariant quantity) is preserved.

Practical takeaway Use gt handoff liberally. Kill sessions without fear. Your agent's identity, inbox, and work history survive because they live in Git, not in the context window.

III. Safe Parallelism Through Isolation

Gas Town achieves parallelism through git worktrees. Each Polecat (worker agent) gets its own isolated copy of the repository. They work simultaneously without interfering — like musicians in separate recording booths, each laying down their track independently.

Mathematically, this is parallel composition: agents compose by working side by side (written ⊗ in category theory), not in sequence (written ∘). The critical property is that the order doesn't matter — whether Agent A finishes before Agent B or vice versa, the Refinery's merge produces the same result. This is the "braiding" symmetry of a monoidal category.

When a merge conflicts — when two agents modify the same file in incompatible ways — you've discovered a failure of the isolation condition. The worktrees weren't truly independent; there was hidden coupling. The fix is architectural: decompose work so that parallel tasks touch different parts of the codebase.

Practical takeaway More worktrees = more safe parallelism, as long as tasks are independent. Merge conflicts aren't random chaos — they're a signal that your task decomposition has hidden coupling.

IV. Don't Boundary the Boundary

In topology, the boundary operator ∂ takes a region and returns its edge. A filled disk becomes a circle. A solid cube becomes its six faces. But here's the deep fact: the boundary of a boundary is always empty. The edge of an edge is nothing. Written: ∂² = 0.

The visualization below lets you apply ∂ step by step. Start with a filled region. Apply ∂ once to get the boundary. Apply it again — you get nothing. This isn't a special case. It's a universal law.

Apply the boundary operator step by step. The boundary of a boundary is always empty: ∂² = 0.

In Gas Town, the Mayor sets the boundary: the project plan, the task decomposition, the architectural decisions. Workers operate within that boundary. If the Mayor — or the Deacon, or the Overseer — redundantly re-specifies context that's already been established, they're computing ∂². The result is zero: no new information, just wasted tokens and contradictory signals.

The infamous "murderous Deacon" bug from January 2026 was precisely this: the Deacon was re-establishing boundaries the Mayor had already set, producing a ∂² ≠ 0 failure that destroyed other workers' coherent state.

Practical takeaway Set context once at the Mayor level. In workers, reference beads — don't re-explain the whole project. Every redundant re-specification risks contradicting the established frame.

V. Coherence: Get the Generators Right

Mac Lane's coherence theorem says: in certain algebraic structures, if a small set of basic operations give the same result regardless of order, then every possible combination of those operations gives the same result. You don't have to check every case — just the generators.

Gas Town has three fundamental protocols — its "generators":

GUPP — if there's work on your hook, you must run it.
Handoff — serialize structural context when a session ends.
Merge — the Refinery lands clean code on main.

If these three protocols are correct, Mac Lane's theorem tells us that all possible multi-agent workflows are coherent — regardless of timing, ordering, or the number of agents. You don't need to test every combination.

The diagram below shows multiple paths through a network of operations. Each path represents a different ordering of agent actions. Toggle between paths — they all arrive at the same final state. That's coherence.

This is also why Gas Town degrades gracefully. Removing a role — disabling the Deacon, the Witness, or the Refinery — is like restricting to a smaller set of operations. The coherence conditions still hold within the subset. You lose functionality, not correctness.

Practical takeaway When things go wrong, check the three protocols — GUPP, handoff, merge — not individual agent behavior. Fix the generator, not every diagram.

These five principles — conservation of work, persistent identity, safe parallelism, ∂²=0, and coherence — aren't engineering heuristics. They're structural invariants. When they hold, Gas Town scales. When they break, the failures are predictable: you can always trace a bug back to a violated conservation law or a broken coherence condition.

Understanding the mathematics doesn't just help you use Gas Town. It helps you understand any multi-agent system — and to predict which ones will work and which will descend into chaos.
