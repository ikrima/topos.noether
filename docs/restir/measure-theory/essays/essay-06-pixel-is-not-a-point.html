Essay 6 · Path-Space Bundle

A Pixel Is Not a Point

A pixel is a coordinate on the screen, but a renderer does not estimate a coordinate. It estimates an integral over a hidden space of paths, hypotheses, shifts, visibility states, and proposal histories attached to that coordinate.

The useful lie: a pixel is a small square of color.

The useful truth: a pixel is a query into a fiber.

p ∈ B = screen × time
above p lives Ωp = local path / hypothesis space
a reservoir is μp ∈ P(Ωp)

The square on the screen is not the object

In a rasterizer, treating a pixel as a little screen square is often enough. In a path tracer with reuse, it becomes dangerously incomplete. The pixel is merely the base location.

What we actually reuse is not a color stored at that location. We reuse a compressed distribution of candidate explanations: light samples, BSDF choices, visibility outcomes, path-shifted samples, and the history of how they arrived.

Click a pixel. The grid point is simple; the fiber above it is not.

base point: p = (i,j,t) fiber: Ωp reservoir: μp

The fiber is where the integral lives

The rendered value at a pixel is not stored in the screen coordinate. It is assembled by integrating an observable over the fiber above that coordinate.

Ip = ∫Ωp fp(x) dμp(x)

In ordinary renderer language: the pixel owns a target function, a proposal history, and a reservoir of weighted representatives. In geometric language: the pixel owns a probability measure on a local path-space fiber.

This distinction matters because neighboring pixels need not share the same fiber. A neighboring pixel can see a different surface, a different light visibility event, or a different path-shift domain.

Reuse is transport between fibers

Spatial reuse is not “copy a neighbor.” Temporal reuse is not “copy the previous pixel.” Both are attempts to map representatives from one fiber to another.

Tq→p: Ωq → Ωp
(Tq→p)# μq = predicted measure in Ωp

When the map stretches, compresses, folds, or partially misses the target fiber, the weights must change. This is the geometric intuition behind GRIS-style shift mappings and Jacobian corrections.

Use the transport type. Notice that the sample cloud itself can remain plausible while the density becomes wrong.

Screen × time is the base; path space is above it

Now assemble the pieces. The visible image sequence is the base space. Above each pixel-time coordinate lives a local fiber of path hypotheses. A reservoir is a tiny empirical measure sitting inside one fiber.

π: Ω → B
B = screen × time
π−1(p) = Ωp

This is why a pixel is not a point. The point is just an address. The renderer’s statistical state lives above the address.

On a flat, stable part of the scene the fibers line up almost trivially. Near visibility changes, specular folds, and motion boundaries, the fibers twist and stop matching.

A transport rule is a connection

A connection tells us how to compare fibers over nearby base points. In renderer terms, it is the rule that says: “this old sample corresponds to this current sample.”

Motion-vector reprojection, object-space transforms, surface-space shifts, and GRIS path shifts are all partial connection rules. They let us lift movement in screen-time into movement inside path space.

base movement: q → p
fiber movement: xq → Tq→p(xq)

If the connection is good, transported reservoirs stay coherent. If it is bad, the transported sample can land in the wrong fiber region with high confidence. That is the geometric origin of ghosting, stale reuse, and many firefly bursts.

What this becomes in a renderer

The bundle view is useful because it turns reuse into a typed operation. You no longer ask whether two pixels are close. You ask whether their fibers can be compared safely.

A reuse candidate should carry both a sample and a claim about transport validity: which fiber it came from, which map carried it, how much density correction it incurred, and how much residual/curvature evidence says to distrust it.

Base coordinatescreen position, time, object ID, material ID, motion state
Fiber statecandidate paths, visibility bits, proposal histories, target values
Connectionmotion vector, path shift, surface shift, learned Jacobian
struct FiberReservoir { Sample sample; float weightSum; float targetPdf; float proposalPdf; uint basePixel; uint frameIndex; uint objectId; uint materialId; uint sourceFiber; uint transportKind; float jacobian; float fiberDistance; float residual; float curvature; float confidence; }; FiberReservoir reuseAcrossFibers(Pixel p, Pixel q) { FiberReservoir Rq = loadReservoir(q); Transport T = chooseConnection(q, p); Sample xp = T.map(Rq.sample); float J = T.jacobian(Rq.sample); float target = evaluateTarget(p, xp); float proposal = evaluateSourceProposal(q, Rq.sample) / max(J, EPS); float mismatch = fiberDistance(q, p, T); float trust = exp(-a*mismatch - b*residual(p) - c*curvature(p)); return reservoirUpdate(xp, target / max(proposal, EPS) * trust); }

The compressed formulation

ReSTIR/GRIS can be read as a calculus over a screen-time bundle of path-space fibers. Pixels are base points. Reservoirs are empirical measures on fibers. Reuse maps are connection-like transports between fibers. Pushforward moves sample measures into the current fiber; pullback moves current tests backward into source fibers. Curvature is the failure of these transports to agree around small screen-time loops.

pixel ≠ point
pixel = base address + hidden fiber + transported empirical measure
