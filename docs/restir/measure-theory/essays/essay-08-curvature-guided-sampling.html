Curvature-Guided Sampling

A renderer should not spend its next samples where the image is bright. It should spend them where transported belief fails.

In the previous essays we made reservoirs visible, transported them between fibers, and measured reuse curvature. Here those concepts become a control loop: predict, compare, gate reuse, and allocate fresh samples.

The renderer as a residual controller

Start with the most boring-looking expression in the whole series. It is also the most practical:

freshSamplePressure = residual + curvature + novelty + entropy + salience

Those terms should not be decorations. Each one answers a different engineering question.

The practical rule: reuse aggressively where transported prediction still explains the evidence; spend new samples where the residual accelerates, curvature rises, or singular novelty appears.

A field of attention, not a scalar heuristic

The quantities above are per-pixel fields. They form a sampling pressure map. A flat wall should cool down. A disocclusion, a shadow boundary, or a glossy fold should heat up.

The reuse valve

Every candidate carries a contribution estimate, but it also carries provenance. The reuse valve converts provenance failure into downweighting, refresh, or extra reservoir cardinality.

reuseTrust = exp(−λ · (aρ + bκ + cν⊥))

Confidence is a contract. A reservoir is allowed to keep an old sample only while the current evidence remains absolutely continuous enough with the transported prior. When the event is singular—a new surface, light toggle, hard visibility flip—the right behavior is not clever weighting. It is refresh.

The pass architecture

The algorithm is intentionally conservative. It does not replace ReSTIR. It inserts a small decision layer around reuse: estimate stability, estimate curvature, gate stale reservoirs, and spend fresh candidates where needed.

struct CurvatureGuidedReservoir { Sample sample; float weightSum; float targetPdf; float confidence; float entropy; float residual; float curvature; float novelty; uint age; uint objectId; uint materialId; }; void updatePixel(Pixel p) { Geometry g = loadGBuffer(p); Reservoir predicted = pushforward(loadPrevReservoir(p), g.motion); Evidence evidence = estimateCurrentEvidence(p, g); float rho = residualNorm(evidence, predicted); float rn = estimateRNRatio(evidence, predicted); float nuP = estimateSingularNovelty(evidence, predicted, g); Reservoir A = temporalThenSpatial(p); Reservoir B = spatialThenTemporal(p); float kappa = reservoirDistance(A, B); float trust = exp(-lambda * (a*rho + b*kappa + c*nuP + abs(log(rn)))); int K = chooseK(baseK, rho, kappa, nuP, entropy(predicted), salience(p)); CandidateSet C = gatherProposals(p, predicted, K); Reservoir R = grisMerge(C, trust, kappa); storeReservoir(p, R); }

What would make this falsifiable?

The best part of the curvature/residual formulation is that it does not have to be believed. It can be instrumented.

A synthetic diagnostic matrix: if the theory is useful, artifact maps should correlate with curvature and singular novelty maps.

Predictions to test

Curvature heatmaps should correlate with ghosting, shimmer, stale shadows, and fireflies.

RN-style reuse gates should outperform binary depth/normal rejection near disocclusions.

Object-space reservoirs should beat screen-space reuse under camera motion and animated objects.

Entropy-aware K should reduce fireflies more efficiently than uniform extra samples.

Semantic proposal channels should help in scenes with structured light/object relationships more than in random geometry.

The final formulation: modern reservoir rendering can be treated as a feedback controller over empirical measures. Prediction is pushforward. Evaluation is pullback. Residual detects error. Curvature detects invalid reuse geometry. Novelty detects singular events. Sampling pressure is where the controller chooses to spend reality checks.
